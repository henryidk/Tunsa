Casos de Uso del Sistema


Módulo 1 — Autenticación y Seguridad
CU-01: Iniciar Sesión
Nombre: Iniciar Sesión
Código: CU-01
Actor principal: Todos los usuarios (Administrador, Secretaria, Colaborador, Encargado de Máquinas)
Descripción: Permite a un usuario autenticarse en el sistema mediante credenciales para acceder a las funciones según su rol.
Precondiciones
El usuario debe tener una cuenta activa en el sistema.
El sistema debe estar disponible.
Flujo Básico
El usuario accede a la pantalla de login.
Ingresa su nombre de usuario y contraseña.
El sistema valida las credenciales contra la base de datos.
El sistema genera un accessToken (15 min) y un refreshToken (7 días).
El sistema registra el evento LOGIN_SUCCESS en el AuditLog con IP y user agent.
Si mustChangePassword = true, redirige al formulario de cambio de contraseña.
El sistema redirige al dashboard correspondiente según el rol.
Flujo Alterno
FA1 – Credenciales inválidas: El sistema muestra un mensaje de error y registra LOGIN_FAILED en el AuditLog.
FA2 – Usuario inactivo: El sistema rechaza el acceso con mensaje de cuenta desactivada.
FA3 – Demasiados intentos: El rate limiter bloquea temporalmente la IP.
Postcondiciones
El usuario queda autenticado con tokens activos en sesión.
Se registra el acceso en el AuditLog.



CU-02: Cerrar Sesión
Nombre: Cerrar Sesión
Código: CU-02
Actor principal: Todos los usuarios autenticados
Descripción: Permite al usuario finalizar su sesión activa, revocando todos sus tokens de seguridad.
Precondiciones
El usuario debe estar autenticado.
Flujo Básico
El usuario hace clic en el botón de cerrar sesión.
El sistema revoca todos los refreshTokens activos del usuario en la base de datos.
El sistema elimina la cookie httpOnly del refreshToken.
El sistema elimina el estado de autenticación del store (Zustand).
El sistema registra LOGOUT en el AuditLog.
El sistema redirige al usuario a la pantalla de login.
Flujo Alterno
FA1 – Error de red: El sistema cierra la sesión localmente aunque falle la revocación en el servidor.
Postcondiciones
La sesión queda invalidada en el servidor.
El usuario es redirigido al login.



CU-03: Cambiar Contraseña (Forzada)
Nombre: Cambiar Contraseña (Forzada)
Código: CU-03
Actor principal: Todos los usuarios
Descripción: Obliga al usuario a establecer una nueva contraseña en su primer acceso o después de un reseteo administrativo, antes de poder usar el sistema.
Precondiciones
El usuario debe haberse autenticado exitosamente.
El campo mustChangePassword debe ser true.
Flujo Básico
El sistema detecta mustChangePassword = true tras el login.
El sistema muestra el modal CambiarPasswordModal.
El usuario ingresa su nueva contraseña y la confirma.
El sistema valida que la nueva contraseña no sea igual a la actual.
El sistema actualiza la contraseña con hash bcrypt.
El sistema establece mustChangePassword = false.
El sistema redirige al dashboard correspondiente al rol.
Flujo Alterno
FA1 – Contraseña igual a la actual: El sistema rechaza el cambio con mensaje de error.
FA2 – Contraseñas no coinciden: El sistema muestra error de validación.
Postcondiciones
La contraseña queda actualizada en la base de datos.
mustChangePassword = false para el usuario.
El usuario accede al sistema.



Módulo 2 — Gestión de Usuarios
CU-04: Crear Usuario
Nombre: Crear Usuario
Código: CU-04
Actor principal: Administrador
Descripción: El administrador registra un nuevo usuario en el sistema asignándole un rol y generando una contraseña temporal segura.
Precondiciones
El actor debe estar autenticado con rol admin.
El username a registrar no debe existir en el sistema.
Flujo Básico
El administrador accede a la sección Usuarios y hace clic en "Agregar usuario".
El sistema muestra el modal AgregarUsuarioModal.
El administrador completa: nombre, username, teléfono (opcional) y rol.
El sistema genera una contraseña segura de 8 caracteres (mayúsculas, minúsculas, dígitos, especiales).
El sistema crea el usuario con mustChangePassword = true e isActive = true.
El modal muestra la contraseña temporal generada para que el administrador la copie.
El sistema muestra un toast de éxito y actualiza la tabla de usuarios.
Flujo Alterno
FA1 – Username duplicado: El sistema muestra error "El nombre de usuario ya existe".
FA2 – Campos inválidos: El sistema muestra mensajes de validación por campo.
Postcondiciones
El usuario queda registrado en la base de datos con rol asignado.
La contraseña temporal debe ser entregada al usuario por el administrador.
El usuario deberá cambiar su contraseña en el primer acceso.



CU-05: Editar Usuario
Nombre: Editar Usuario
Código: CU-05
Actor principal: Administrador
Descripción: El administrador modifica los datos personales (nombre, username, teléfono) de un usuario existente.
Precondiciones
El actor debe estar autenticado con rol admin.
El usuario a editar debe existir y ser diferente al usuario actual (no puede editarse a sí mismo).
Flujo Básico
El administrador localiza el usuario en la tabla y hace clic en "Editar".
El sistema muestra el modal EditarUsuarioModal con los datos actuales.
El administrador modifica los campos deseados.
El sistema valida los datos y actualiza el registro.
El sistema muestra un toast de éxito y refresca la tabla.
Flujo Alterno
FA1 – Username ya en uso: El sistema muestra error de unicidad.
FA2 – Sin cambios: El sistema acepta la solicitud sin modificar nada.
Postcondiciones
Los datos del usuario quedan actualizados en la base de datos.



CU-06: Desactivar Usuario
Nombre: Desactivar Usuario
Código: CU-06
Actor principal: Administrador
Descripción: El administrador desactiva la cuenta de un usuario, impidiendo su acceso al sistema e invalidando todas sus sesiones activas.
Precondiciones
El actor debe estar autenticado con rol admin.
El usuario a desactivar debe estar activo y ser diferente al usuario actual.
Flujo Básico
El administrador hace clic en "Desactivar" junto al usuario.
El sistema muestra el modal de confirmación ConfirmDesactivarModal.
El administrador confirma la acción.
El sistema establece isActive = false en el usuario.
El sistema revoca todos los refreshTokens activos del usuario.
El sistema muestra un toast de éxito.
El estado del usuario en la tabla cambia a "Inactivo".
Flujo Alterno
FA1 – El administrador cancela: No se realiza ningún cambio.
Postcondiciones
El usuario no puede iniciar sesión.
Sus sesiones activas son invalidadas inmediatamente.



CU-07: Activar Usuario
Nombre: Activar Usuario
Código: CU-07
Actor principal: Administrador
Descripción: El administrador reactiva la cuenta de un usuario previamente desactivado, permitiéndole volver a iniciar sesión.
Precondiciones
El actor debe estar autenticado con rol admin.
El usuario a activar debe estar en estado inactivo.
Flujo Básico
El administrador localiza al usuario inactivo en la tabla.
Hace clic en el botón "Activar".
El sistema establece isActive = true.
El sistema muestra un toast de éxito y actualiza la tabla.
Flujo Alterno
FA1 – Error del servidor: El sistema muestra toast de error y revierte el estado visual.
Postcondiciones
El usuario puede volver a iniciar sesión con sus credenciales actuales.



CU-08: Resetear Contraseña de Usuario
Nombre: Resetear Contraseña de Usuario
Código: CU-08
Actor principal: Administrador
Descripción: El administrador restablece la contraseña de un usuario, generando una nueva temporal y revocando todas sus sesiones activas.
Precondiciones
El actor debe estar autenticado con rol admin.
El usuario objetivo debe existir en el sistema.
Flujo Básico
El administrador hace clic en "Resetear" junto al usuario.
El sistema muestra el modal ResetPasswordModal.
El sistema genera una nueva contraseña segura de 8 caracteres.
El sistema actualiza la contraseña con hash bcrypt.
El sistema establece mustChangePassword = true.
El sistema revoca todos los refreshTokens activos del usuario.
El sistema registra PASSWORD_RESET_BY_ADMIN en el AuditLog.
El modal muestra la nueva contraseña temporal.
Flujo Alterno
FA1 – Error de servidor: El sistema muestra toast de error.
Postcondiciones
La contraseña del usuario queda actualizada.
El usuario deberá cambiar su contraseña en su próximo acceso.
Todas las sesiones previas del usuario son invalidadas.



Módulo 3 — Gestión de Equipos
CU-09: Agregar Equipo al Inventario
Nombre: Agregar Equipo al Inventario
Código: CU-09
Actor principal: Administrador, Encargado de Maquinas, Secretaria
Descripción: El administrador registra un nuevo equipo en el inventario con sus datos técnicos y precios de renta.
Precondiciones
El actor debe estar autenticado con rol admin.
La numeración del equipo no debe existir previamente.
Flujo Básico
El administrador accede a la sección Equipos y abre AgregarEquipoModal.
Completa: numeración, descripción, categoría, serie (opcional), cantidad, fecha de compra, monto de compra, tipo (Liviana / Pesada / Uso Propio).
Opcionalmente ingresa precios de renta por día, semana y mes.
El sistema valida los datos y crea el registro con isActive = true.
El sistema muestra toast de éxito y actualiza la tabla.
Flujo Alterno
FA1 – Numeración duplicada: El sistema rechaza con mensaje de error.
FA2 – Campos obligatorios vacíos: El sistema muestra validaciones por campo.
Postcondiciones
El equipo queda registrado en el inventario como activo.
Los valores del inventario (total Q) se actualizan en las estadísticas.



CU-10: Editar Datos de Equipo
Nombre: Editar Datos de Equipo
Código: CU-10
Actor principal: Administrador, Encargado de maquinas, Secretaria
Descripción: El administrador modifica los datos técnicos de un equipo existente, quedando registrado el historial de cambios en la bitácora.
Precondiciones
El actor debe estar autenticado con rol admin.
El equipo debe estar activo en el inventario.
Flujo Básico
El administrador hace clic en "Editar" en la fila del equipo.
El sistema muestra EditarEquipoModal con los datos actuales.
El administrador modifica los campos deseados.
El sistema calcula las diferencias campo por campo.
Para cada campo modificado, el sistema genera una entrada en la Bitácora con valor anterior y nuevo.
El sistema actualiza el registro del equipo.
El sistema muestra toast de éxito.
Flujo Alterno
FA1 – Sin cambios: El sistema no genera entradas en bitácora.
FA2 – Numeración duplicada: El sistema rechaza el cambio.
Postcondiciones
Los datos del equipo quedan actualizados.
Cada campo modificado queda registrado en la Bitácora con quién realizó el cambio.



CU-11: Actualizar Precios de Renta de Equipo
Nombre: Actualizar Precios de Renta de Equipo
Código: CU-11
Actor principal: Administrador, Secretaria
Descripción: El administrador actualiza los precios de renta (día, semana, mes) de un equipo, con registro en bitácora.
Precondiciones
El actor debe estar autenticado con rol admin.
El equipo debe existir y estar activo.
Flujo Básico
El administrador hace clic en "Precios" en la fila del equipo.
El sistema muestra PreciosEquipoModal con los precios actuales.
El administrador actualiza uno o más precios.
El sistema genera entradas en Bitácora para cada precio modificado.
El sistema actualiza los precios y muestra toast de éxito.
Flujo Alterno
FA1 – Precios negativos: El sistema muestra error de validación.
Postcondiciones
Los nuevos precios se aplican a futuras solicitudes de renta.
Los cambios quedan registrados en Bitácora.



CU-12: Dar de Baja Equipo
Nombre: Dar de Baja Equipo
Código: CU-12
Actor principal: Administrador, Secretaria
Descripción: El administrador retira un equipo del inventario activo indicando el motivo, sin eliminarlo del sistema.
Precondiciones
El actor debe estar autenticado con rol admin.
El equipo debe estar activo.
Flujo Básico
El administrador hace clic en "Dar de baja" en la fila del equipo.
El sistema muestra BajaEquipoModal.
El administrador ingresa el motivo de la baja.
El sistema establece isActive = false, registra motivoBaja y fechaBaja.
El sistema genera entrada en Bitácora.
El equipo se mueve a la pestaña "Dados de baja".
Flujo Alterno
FA1 – Motivo vacío: El sistema requiere ingresar motivo antes de confirmar.
Postcondiciones
El equipo queda en estado inactivo con fecha y motivo de baja.
Aparece en la pestaña "Dados de baja" y no en el inventario activo.



CU-13: Reactivar Equipo
Nombre: Reactivar Equipo
Código: CU-13
Actor principal: Administrador
Descripción: El administrador reactiva un equipo previamente dado de baja, reincorporándolo al inventario activo.
Precondiciones
El actor debe estar autenticado con rol admin.
El equipo debe estar en estado de baja (isActive = false).
Flujo Básico
El administrador accede a la pestaña "Dados de baja" en Equipos.
Hace clic en "Reactivar" en la fila del equipo (muestra spinner durante la operación).
El sistema establece isActive = true y limpia motivoBaja y fechaBaja.
El sistema genera entrada en Bitácora.
El equipo se mueve a la pestaña "Activos".
Flujo Alterno
FA1 – Error de servidor: El sistema revierte el estado visual y muestra error.
Postcondiciones
El equipo queda disponible en el inventario activo.



CU-14: Generar Reporte PDF de Inventario
Nombre: Generar Reporte PDF de Inventario de Equipos
Código: CU-14
Actor principal: Administrador, Secretaria, Encargado de maquinas
Descripción: El administrador genera y descarga un reporte PDF con el inventario completo de equipos, organizado por tipo con valores y estadísticas.
Precondiciones
El actor debe estar autenticado con rol admin.
Deben existir equipos registrados en el sistema.
Flujo Básico
El administrador hace clic en "Generar reporte" en la sección Equipos.
El sistema recopila todos los equipos de la base de datos.
El sistema genera el PDF en formato A4 landscape con: tarjetas de resumen (conteo y valor por tipo), tablas por sección (Liviana, Pesada, Uso Propio, Dados de Baja), encabezados y pies de página con número de página, y formato de moneda (Q).
El sistema descarga el archivo como inventario_equipos_YYYY-MM-DD.pdf.
Flujo Alterno
FA1 – Sin equipos: El sistema genera un reporte vacío o con mensaje informativo.
Postcondiciones
El archivo PDF queda descargado en el dispositivo del usuario.



Módulo 4 — Rentas y Clientes
CU-15: Registrar Cliente
Nombre: Registrar Cliente
Código: CU-15
Actor principal: Administrador, Secretaria, Encargado de maquinas
Descripción: El actor registra un nuevo cliente en el sistema con sus datos de identificación para gestionar sus rentas.
Precondiciones
El actor debe estar autenticado con rol admin o secretaria.
El DPI del cliente no debe estar registrado previamente.
Flujo Básico
El actor accede a la sección Clientes y hace clic en "Registrar cliente".
El sistema muestra RegistrarClienteModal.
El actor ingresa: nombre, DPI, teléfono (opcional).
El sistema genera automáticamente un código único con formato CLI-XXXX.
El sistema crea el registro del cliente.
El sistema muestra toast de éxito y actualiza la tabla.
Flujo Alterno
FA1 – DPI duplicado: El sistema muestra error de DPI ya registrado.
FA2 – Nombre o DPI vacíos: El sistema muestra validaciones.
Postcondiciones
El cliente queda registrado con código único asignado.
Puede ser asociado a solicitudes de renta.



CU-16: Editar Cliente
Nombre: Editar Cliente
Código: CU-16

Precondiciones

CU-18: Crear Solicitud de Renta
Nombre: Crear Solicitud de Renta
Código: CU-18
Actor principal: Encargado de Máquinas
Descripción: El encargado de máquinas crea una solicitud de renta de equipos para un cliente, la cual queda pendiente de aprobación.
Precondiciones
El actor debe estar autenticado con rol encargado_maquinas.
Deben existir clientes y equipos disponibles registrados en el sistema.
Flujo Básico
El actor accede a Solicitudes de Renta y crea una nueva solicitud.
Selecciona el cliente de la lista.
Agrega uno o más equipos con su período de renta.
El sistema calcula el total estimado basado en los precios de renta.
El actor confirma la solicitud.
El sistema crea la solicitud con estado Pendiente y solicitadoPor = usuario actual.
La solicitud aparece en el dashboard del administrador con badge "pendiente".
Flujo Alterno
FA1 – Sin equipos seleccionados: El sistema requiere al menos un equipo.
FA2 – Equipo ya en renta: El sistema muestra advertencia de disponibilidad.
Postcondiciones
La solicitud queda registrada con estado Pendiente.
El administrador y la secretaria pueden visualizarla para aprobar o rechazar.



CU-19: Aprobar Solicitud de Renta
Nombre: Aprobar Solicitud de Renta
Código: CU-19
Actor principal: Administrador, Secretaria
Descripción: El actor revisa y aprueba una solicitud de renta pendiente, activando el contrato de renta.
Precondiciones
El actor debe estar autenticado con rol admin o secretaria.
La solicitud debe existir con estado Pendiente.
Flujo Básico
El actor accede a la sección Solicitudes y filtra por "Pendientes".
Selecciona una solicitud y hace clic en "Ver detalle".
El sistema muestra RentaModal con: cliente, equipos, período, total estimado.
El actor revisa los datos y hace clic en "Aprobar".
El sistema cambia el estado a Aprobada.
Los equipos involucrados se marcan como en renta.
La renta aparece en la sección Rentas Activas.
El sistema muestra toast de éxito.
Flujo Alterno
FA1 – Equipo ya no disponible: El sistema muestra advertencia.
Postcondiciones
La solicitud queda con estado Aprobada.
El contrato aparece en Rentas Activas con el conteo de días.

Nombre: Rechazar Solicitud de Renta
Código: CU-20

Postcondiciones
La solicitud queda con estado Rechazada.
Aparece en el historial de solicitudes rechazadas.

Nombre: Ver Rentas Activas

CU-22: Gestionar Renta Vencida
Nombre: Gestionar Renta Vencida
Código: CU-22
Actor principal: Administrador, Secretaria, Encargado de maquinas
Descripción: El administrador gestiona las rentas que han superado su fecha de vencimiento,
Precondiciones
El actor debe estar autenticado con rol admin.
Deben existir rentas con fecha de vencimiento superada.
Flujo Básico
El actor accede a la sección Rentas Vencidas.
El sistema muestra el banner de alerta con el total de rentas vencidas.
Para cada renta vencida se muestra: contrato, cliente con teléfono, equipos, fecha de vencimiento, días vencida y recargo estimado.
